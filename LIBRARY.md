[L0003] New host-side tool targets hit the same MSVC portability traps — check before pushing | candidate | added: 2026-07-18 | tags: plugin-platform | lesson: CI builds every tool target on windows-latest/MSVC, and each new add_executable repeats the same three traps: (1) no M_PI without _USE_MATH_DEFINES — use a local kPi constant (bit twice in one day: renderer_bench, then trajectory_check); (2) -O3 is not an MSVC flag — guard target_compile_options with if(NOT MSVC); (3) the project links the static MSVC runtime project-wide (CMAKE_MSVC_RUNTIME_LIBRARY at root — clap-wrapper only sets it in its own scope, and mixing /MT with /MD is an LNK2038 wall). [L0004] A filed PR's branch is frozen — pushing follow-ups races the human's merge | candidate | added: 2026-07-18 | tags: plugin-platform | lesson: Twice in two days a commit missed its merge window: PR #4 merged into a stacked base after the base's own PR had merged (stranding the GUI off main), and a layout+ADSR commit pushed to PR #5's branch landed minutes after the human merged it. The human merges asynchronously — any push to an already-filed PR branch is a race. Rule: once a PR is filed, its branch is frozen; new work starts a NEW branch (even for "the same topic"), and stacked PRs state their merge order in the body. After any merge, verify the expected commit is an ancestor of origin/main (git merge-base --is-ancestor) before building on top. | evidence: PR #4 stranding (traces/2026-07-18-gui-polish.md) and PR #5/#6 miss (this session) — both required rescue PRs. | falsifier: a repo setting or workflow (e.g. auto-merge with required approvals) makes post-file pushes safe, or the human starts announcing merge timing. | supersedes: —
